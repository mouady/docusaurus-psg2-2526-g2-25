## 1. Coding standards

### 1.1. Organización visual de los archivos

Para que nuestro código se lea de forma fluida y todos sintamos que trabajamos en el mismo proyecto, vamos a mantener una estructura visual común. Simplemente dejaremos un espacio en blanco entre los `imports` y el inicio de la clase. Una vez dentro, dejamos otro espacio antes de poner los atributos, los cuales irán todos juntos sin líneas en blanco entre ellos. Después, a la hora de escribir los métodos, sí que dejaremos un espacio entre cada uno para que el código respire y sea más fácil de repasar.

```java
import java.util.List;
import java.util.UUID;

public class PatientService {

    private String patientId;
    private String fullName;
    private int age;

    public void registerPatient() {
        // Lógica de registro
    }

    public void updatePatient() {
        // Lógica de actualización
    }
}
```

### 1.2. Estructuras de control

Cuando usemos condicionales o bucles (como `if`, `else` o `for`), la regla es muy sencilla: abrimos la llave `{` en la misma línea donde hacemos la comprobación. El código de dentro empieza directamente en la línea de abajo, usando la tabulación normal del sistema, sin dejar espacios extra.

```java
if (patientAge >= 18) {
    processAdultPatient();
} else {
    processMinor();
}

for (int i = 0; i < maxCapacity; i++) {
    assignRoomToPatient();
}
```

### 1.3. Nombres y conexión entre Backend y Frontend

Para los nombres, usaremos siempre *PascalCase* en las clases y DTOs, recordando dejar las siglas en mayúsculas (por ejemplo, `ClinicDTO`). Para los atributos, la norma es usar *camelCase*. Lo más importante aquí es que esta misma regla de *camelCase* la apliquemos exactamente igual en React. Así nos aseguramos de que el backend y el frontend hablan el mismo idioma y no perdemos datos por el camino.

```java
// En nuestro backend (Java)
public class ClinicDTO {
    private String clinicAddress;
    private int maxCapacity;
}
```

```javascript
// En nuestro frontend (React)
const clinicData = {
    clinicAddress: "Calle Falsa 123",
    maxCapacity: 50
};
```

### 1.4. Enums y anotaciones

Para tener todo el contexto a mano, vamos a declarar los *Enums* dentro de la propia entidad a la que pertenecen, en lugar de crearles archivos separados. Visualmente, las anotaciones (como `@Id` o `@Entity`) las pondremos en formato "torre": una debajo de otra, ocupando una sola línea cada una, justo encima de lo que estén modificando. 

```java
@Entity
@Table(name = "appointments")
@Data
public class Appointment {

    public enum AppointmentStatus {
        PENDING, CONFIRMED, CANCELLED
    }

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "id")
    private Long id;
    
    @Enumerated(EnumType.STRING)
    private AppointmentStatus status;
}
```

### 1.5. Cómo hacemos los Tests

Finalmente, para que las pruebas no se conviertan en un dolor de cabeza, es vital que todos sigamos la misma estructura a la hora de testear. Básicamente, seguiremos el patrón que ya tenemos en el proyecto: separamos claramente la parte donde preparamos los datos, la acción que queremos probar y, por último, la comprobación de que todo ha ido bien.

```java
@Test
void shouldRegisterNewPatientSuccessfully() {
    // 1. Preparamos los datos
    PatientDTO patient = new PatientDTO("Juan", "Perez");

    // 2. Ejecutamos la acción
    boolean isRegistered = patientService.register(patient);

    // 3. Comprobamos el resultado
    assertTrue(isRegistered);
}
```

## 2. Políticas de mensajes de commit

Nuestra política de mensajes de commit estará basada en [Conventional Commits 1.0.0](https://www.conventionalcommits.org/en/v1.0.0/). A continuación, se describe la política a usar en nuestros mensajes de commits. 

### Estructura
 
```
<type>(<optional scope>): <description>

<optional body>

<optional footer>
```

A partir de este esquema general, profundizaremos en cada elemento del mensaje de commit:

**1. Tipo (`<type>`):**
Hace referencia al tipo de commit que podemos realizar sobre el repositorio. Podemos diferenciar entre los tipos: 
- `feat`: introduce una nueva feature al código base. 
- `docs`: introduce nuevos informes o cambios en los mismos. 
- `fix`: introduce arreglo de bug en el código.
- `refactor`: introduce refactorización sobre determinado código.
- `test`: introduce pruebas.
- `style`: introduce cambios de estilo para la aplicación web.

**2. Scope (`<optional scope>`):**

Es opcional y hace referencia al trozo de código afectado en el commit.

**3. Descripción (`<description>`):**

Resumen corto en tiempo presente. 

**4. Cuerpo (`<optional body>`):**

Explicación opcional más detallada.

**5. Footer (`<optional footer>`):**

Campo opcional que podemos usar para:

- BREAKING CHANGE: `<description>`. 
- Fixes #issue: para cerrar la issue.
- Refs #issue: para referenciar otra issue. 
 
---

### Ejemplos prácticos

**Commit básico con tipo y descripción:**
```
feat: add pet appointment module
```

**Commit con scope:**
```
fix(api): resolve null pointer exception in getPetById
```

**Commit con cuerpo explicativo:**
```
refactor(database): optimize query performance

Changed the database queries to use indexed fields
and reduced the number of joins, improving response
time by approximately 40%.
```

**Commit con breaking change:**
```
feat(auth)!: change authentication token format

BREAKING CHANGE: The authentication token format has been changed
from JWT to a custom format. All existing tokens will be invalidated
and users will need to log in again.
```

**Commit que cierra una issue:**
```
fix(login): correct password validation logic

The password validation was not checking for special
characters correctly, allowing weak passwords.

Fixes #42
```

**Commit que referencia múltiples issues:**
```
docs: update API documentation

Added examples for all endpoints and updated
parameter descriptions.

Refs #15, #23, #31
```

---

### Reglas y buenas Prácticas

1. **Descripción:**
   - Debe estar en minúsculas.
   - No debe terminar en punto.
   - Máximo 70 caracteres.
   - Usar imperativo presente. 

2. **Cuerpo:**
   - Separar del título con una línea en blanco.
   - Explicar el "qué" y el "por qué", no el "cómo".
   - Máximo 70 caracteres por línea.

3. **Breaking Changes:**
   - Añadir `!` después del tipo/scope: `feat!:` o `feat(api)!:`
   - Siempre incluir `BREAKING CHANGE:` en el footer con explicación

4. **Scopes sugeridos para nuestro proyecto:**
   - `auth`: autenticación y autorización.
   - `api`: endpoints y controladores,
   - `database`: modelos y migraciones,
   - `ui`: componentes de interfaz,
   - `test`: pruebas unitarias e integración.
   - `docs`: documentación.

---

### Commits NO permitidos

#### **Evitar:**
```
Update file
Fixed bug
WIP
asdfgh
Merge branch 'main'
```

#### **Correcto:**
```
fix(auth): prevent login with invalid credentials
feat(user): add profile picture upload
docs(readme): update installation steps
```

---

### Verificación de Commits

Antes de hacer push, verifica que tu commit cumple:

- Tiene un tipo válido (`feat`, `fix`, `docs`, etc.).
- La descripción es clara y concisa.
- Usa imperativo presente.
- No excede 70 caracteres en el título.
- Si hay breaking changes, está marcado correctamente.
- Referencias a issues son correctas.

## Estructura del repositorio y ramas por defecto

### 3.1 Estructura del repositorio
Para mantener una arquitectura limpia, escalable y manejable, nuestro proyecto utiliza un enfoque de repositorio único que engloba tanto el código del *frontend* como del *backend*, junto con las configuraciones de despliegue y la documentación. 

En la raíz del repositorio, la estructura principal se divide en tres grandes bloques: la carpeta **`docs/`**, que contiene toda la documentación generada durante el ciclo de vida del proyecto (incluyendo este informe técnico en formato Markdown para facilitar el control de versiones, actas de reuniones y políticas); el directorio **`frontend/`**, que aloja el código fuente y los recursos de la aplicación cliente; y, finalmente, la carpeta **`src/`**, destinada al código fuente principal del *backend* desarrollado en Java.

Además de estos directorios, la raíz contiene varios archivos de configuración y *scripts* esenciales para la gestión automatizada del proyecto:
* **Gestión de dependencias:** Empleamos **`pom.xml`** junto con los *scripts* de Maven Wrapper (**`mvnw`**, **`mvnw.cmd`** y la carpeta oculta **`.mvn/`**). Esto garantiza que todos los miembros del equipo utilicen exactamente la misma versión de Maven sin necesidad de instalaciones locales previas.
* **Entornos y despliegue:** La orquestación de contenedores para levantar el entorno de desarrollo de forma consistente se define en el archivo **`docker-compose.yml`**.
* **Estandarización y seguridad:** Para asegurar la consistencia en el formato del código entre diferentes editores (sangrías, codificación, etc.), utilizamos **`.editorconfig`**. Por su parte, los archivos **`.gitignore`** y **`.gcloudignore`** son fundamentales para prevenir la subida de archivos binarios, dependencias locales o credenciales sensibles al control de versiones o a los servidores de Google Cloud.
* **Información general:** El punto de entrada del repositorio es el archivo **`README.md`**, acompañado de un archivo **`info.yml`**.

### 3.2 Ramas por defecto
Dado que nuestra metodología de gestión de la configuración se basa estrictamente en **Git Flow**, el repositorio se articula en torno a dos ramas principales de larga duración que dictan nuestro flujo de trabajo base:

1. **`main` (Rama de producción):** Esta rama almacena el historial de versiones oficiales y estables del proyecto. El código alojado aquí debe estar siempre en un estado funcional y listo para su paso a producción. Para proteger su integridad, los *commits* directos a `main` están estrictamente prohibidos; cualquier actualización debe integrarse obligatoriamente mediante Pull Requests que vengan de `develop`.
2. **`develop` (Rama de integración):** Funciona como la rama central para el trabajo diario de los desarrolladores. Es el entorno de integración donde se unifica el trabajo de todas las nuevas características (*feature branches*) una vez que han sido terminadas y revisadas por otros miembros del equipo.

## 4. Branching strategy

### Ramas principales (lifetime indefinido)

El proyecto seguirá **Git Flow** como metodología de ramificación, apoyándonos en dos ramas principales y de vida indefinida: `main` y `develop`.

- **`main`**: rama estable y siempre lista para producción.  
- **`develop`**: rama de integración con los cambios destinados a la próxima release.

### Ramas de soporte (supporting branches)

Además, se utilizarán ramas de soporte propias de Git Flow:

- **`feature/*`**: desarrollo de nuevas funcionalidades.  
- **`release/*`**: preparación de una versión de producción.  
- **`hotfix/*`**: corrección de errores críticos detectados en producción.
- **`fix/*` (convención del equipo)**: corrección de bugs detectados durante el desarrollo (sale de `develop` y vuelve a `develop` vía PR).
- **`docs/*` (convención del equipo)**: creación o actualización de documentación del proyecto (sale de `develop` y vuelve a `develop` vía PR).

### Peer reviews obligatorias (Pull Requests)

Todo cambio de configuración/código debe ser revisado por al menos un miembro antes de integrarse, en tareas de mayor complejidad se podrán 
asignar varios miembros, usando **peer reviews** mediante **Pull Requests**. Esto mejora el entendimiento compartido, refuerza estándares, aumenta la calidad y reduce el "bus factor".

### Adaptación obligatoria al proyecto (uso de GitHub)

En este proyecto, Git Flow se aplica con una modificación operativa:

- La creación de ramas y las fusiones se harán obligatoriamente en **GitHub** mediante **Pull Requests**.  
- Todas las ramas deben crearse desde GitHub usando **"Create branch from issue"** y quedar vinculadas a una **Issue** (feature, fix, refactor…).  
- No se permiten merges directos: toda integración en `develop` o `main` debe pasar por PR.  
- Los comandos locales de Git Flow se consideran solo informativos y no la forma principal de trabajar en el proyecto.

## a. How to develop feature branches

### Principio general

Las feature branches deben ser **cortas (short-lived)** y orientadas a incrementos pequeños. El objetivo es integrar a menudo y facilitar la revisión por PR.

### Flujo de trabajo (adaptado al proyecto)

Proceso para desarrollar una feature:

1. Crear una **Issue** que describa la funcionalidad.  
2. Crear la rama desde GitHub con **Create branch from issue** y asociarla a esa Issue.  
3. Nombrado recomendado: `feature/<id-issue>-<breve-descripcion>`.  
4. La rama `feature/*` sale de `develop` (base de la próxima release) y se publica en remoto (GitHub) para trazabilidad.

### Revisión e integración (PR)
Al cerrar la PR, la integración debe evitar "fast-forward" y dejar un **merge commit** (equivalente a `--no-ff`) para que quede visible el historial de la feature y sea más fácil auditar/revertir.

## b. How to prepare releases

### Objetivo de las release branches

Las ramas `release/*` se usan para preparar una nueva versión de producción. En esta fase solo se permiten:
- cambios de última hora,
- bug fixes menores,
- y metadatos de release (p. ej., documentación/versión).

### Flujo de release (Git Flow)

Proceso estándar:

1. Crear `release/<version>` desde `develop` cuando el conjunto de features planificado esté listo.  
2. Estabilizar la release con correcciones menores.  
3. Integrar la release en `main` y crear un **tag de versión** asociado a esa entrega.  
4. Integrar también `release/*` de vuelta a `develop` para no perder los ajustes de estabilización.

## c. How to fix bugs

### Bugs en desarrollo 

Los bugs detectados en desarrollo se corrigen como trabajo normal del proyecto:
- se crea una **Issue** del bug,
- se abre una rama vinculada a esa Issue (por ejemplo `fix/*`) partiendo de `develop`,
- y se integra de vuelta a `develop` mediante **Pull Request** con **peer review**.

### Hotfix: bug crítico en producción (ya desplegado)

Si el bug es crítico y está en producción, se usa una rama `hotfix/*` partiendo de `main`, para restaurar rápidamente un estado estable.

Flujo Git Flow para hotfix:

1. Abrir una Issue del bug crítico y crear `hotfix/<version>` desde `main`.  
2. Implementar la corrección con cambios mínimos y verificables.  
3. Integrar en `main` y etiquetar la nueva versión (normalmente incremento **patch**).  
4. Integrar también en `develop` para que la corrección quede incluida en el trabajo futuro (salvo que haya una release en curso).

Igual que con el resto del proyecto, se trabaja con ramas ligadas a Issues y PR para integraciones.

## 5. Políticas de versionado

El proyecto sigue una estrategia de versionado basada en ***semantic versioning***:

**X.Y.Z**

Donde:

- **X**: cambios incompatibles con versiones anteriores o modificaciones estructurales importantes, que rompen la compatibilidad con la API. Aplicado a este proyecto, salvo en algún caso excepcional, sólo se cambiará la etiqueta **X** cuando se cambie de sprint.
- **Y**: nuevas funcionalidades compatibles con versiones anteriores, cambios menores, o funcionalidades nuevas y mejoras.
- **Z**: corrección de errores (*bug fixes*), sólo cambios internos para corregir comportamientos incorrectos de la aplicación.

> Nota: cuando **X** aumente, se resetearán (volverán a 0) las demás; y cuando **Y** aumente, se reseteará **Z**.

## 6. Definición de "hecho"

Una tarea se considera hecha cuando cumple todos los siguientes criterios, en ningún orden concreto:

- El código ha sido revisado y aprobado mediante una *pull request* por al menos otro miembro del equipo.
- Todas las pruebas unitarias y de integración relacionadas con el cambio pasan correctamente, cubriendo al menos el 75% del código afectado.
- La funcionalidad ha sido probada en el entorno de desarrollo y no provoca errores en dicho entorno.
- La documentación técnica asociada está completa y actualizada. Se debe especificar explícitamente en la descripción de la tarea o commit qué documentación se ve afectada por el cambio (por ejemplo: comentarios en clases `X`, documentación de la API en archivos como `docs/api.md`, diagramas de arquitectura, etc.).
- No existen errores de compilación tras la implementación de la nueva funcionalidad.
- La implementación sigue los estándares de codificación y estilo acordados por el equipo.
- Cualquier cambio relevante en la base de datos, API o integraciones externas ha sido documentado y testeado.  
- La *pull request* correspondiente a la tarea ha sido revisada y aceptada por el revisor asignado, de acuerdo al sistema de *peer reviews* definido por el equipo, siguiendo el orden alfabético de los miembros para asignar al revisor de cada *pull request*.

## 7. Manejo de documentación en el repositorio

En este repositorio se ha establecido la siguiente política de gestión de documentos:

- 📂 **commitment-agreements** : los commitment agreements en pdf firmados de cada uno de los integrantes
- 📂 **reports**: los technical reports que se describen en el backlog.
- 📂 **actas**
    - 📂 **Sprint 1**
    - 📂 **Sprint 2**
    - 📂 **Sprint 3**
    - 📂 **Sprint 4**

Por cada reunion se generará AL MENOS:

- 📂 **DD-MM-tematica**
    - 📄 raw-DD-MM-tematica-autor.md
    - 📄 DD-MM-tematica.md

**Sobre el concepto de raw:** el raw es un borrador en formato markdown que se redacta durante la reunion y que posteriormente se utiliza (no se borra, ni se edita) para generar el acta final. Se pueden generar varios raws por cada autor si es necesario, pero solo hay UN acta final por cada reunion que tiene caracter oficial.

Donde:
- **tematica**: es la temática de la reunión. Los valores posibles son:

| Valor | Cuándo usarlo |
|---|---|
| `planning` | Reunión de planificación del sprint. |
| `standup` | Reuniones que se mantienen EN clase de laboratorio. |
| `daily` | Reuniones que se mantienen FUERA de clase de laboratorio. |
| `review` | Reunión de revisión al final del sprint. Se presenta el trabajo completado a los interesados y se recoge feedback. |
| `retrospective` | Reunión de retrospectiva al final del sprint. El equipo reflexiona sobre su forma de trabajar para identificar mejoras. |
- **DD-MM**: es la fecha de la reunion. Ejemplo: 01-10 (1 de octubre).
- **autor**: es el nombre del integrante que ha redactado lo que llamamos el "raw" del acta.

### Uso de plantillas para actas oficiales

Siempre que sea posible, el acta oficial (`DD-MM-tematica.md`) deberá generarse utilizando la plantilla correspondiente disponible en `docs/templates/`. El uso de estas plantillas es obligatorio para garantizar la uniformidad y coherencia de toda la documentación oficial del proyecto.

## 8. Informe de iTop y del CMDB

* **Sistemas de cómputo:** Ordenadores personales, portátiles y dispositivos móviles.

### 8.3. Relaciones y análisis de dependencias

Un aspecto fundamental de la estructura del CMDB es la vinculación lógica entre los elementos. En nuestra implementación, se han establecido las siguientes relaciones clave:
